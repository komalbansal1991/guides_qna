"When should I create a new baseline vs. a new version in AEM Guides?"
"

In AEM Guides, the decision to create a new baseline or a new version depends on the purpose and scope of the changes you are managing in your DITA content.

When to Create a New Baseline:

  • Purpose: A baseline is used to capture a snapshot of your content at a specific point in time, typically for release management or archival purposes.
  • Use Cases:
    • When preparing for a product release or milestone.
    • To freeze content for a specific version of a product or documentation set.
    • For compliance or regulatory requirements where you need to maintain a record of the content as it was at a specific time.
  • Key Feature: Baselines are static and do not change even if the source content is updated.

When to Create a New Version:

  • Purpose: A version is used to track changes to individual files or maps over time, allowing you to revert to previous states if needed.
  • Use Cases:
    • When making iterative updates to content.
    • To track changes made by different authors or during collaborative editing.
    • For maintaining a history of edits to a specific file or map.
  • Key Feature: Versions are dynamic and tied to the version history of the content.

Summary:

  • Use baselines for release management and archival purposes.
  • Use versions for tracking changes and maintaining a history of edits.

"What are common validation patterns for technical documentation?"
"

Common validation patterns for technical documentation in AEM Guides often involve using Schematron rules to ensure content quality and consistency. These patterns can be applied to DITA topics or maps to enforce specific standards.

Examples of Common Validation Patterns:

  1. External Links Validation:

    • Ensure all external links have the correct scope and format attributes.
      xml <sch:rule context="xref[contains(@href, 'http') or contains(@href, 'https')]"> <sch:assert test="@scope = 'external' and @format = 'html'"> All external xref links must have scope='external' and format='html'. </sch:assert> </sch:rule>

  2. Mandatory Elements:

    • Check if a topicref exists in a map or if a ul contains at least one li.
      xml <sch:rule context="map"> <sch:assert test="count(topicref) > 0"> There should be at least one topicref in the map. </sch:assert> </sch:rule>

  3. Element Placement:

    • Ensure indexterm elements are always within a prolog.
      xml <sch:rule context="*[contains(@class, ' topic/indexterm ')]"> <sch:assert test="ancestor::node()/local-name() = 'prolog'"> The indexterm element should be in a prolog. </sch:assert> </sch:rule>

  4. Character Length Validation:

    • Validate that a short description does not exceed a specific character limit.
      xml <sch:rule context="shortdesc"> <sch:let name="characters" value="string-length(.)"/> <sch:report test="$characters > 150"> The short description has <sch:value-of select="$characters"/> characters. It should not exceed 150 characters. </sch:report> </sch:rule>

  5. Regex-Based Validation:

    • Use regex to enforce specific patterns, such as disallowing one-word titles.
      xml <assert test="not(matches(.,'^\w+$'))"> No one-word titles. </assert>

  6. Abstract Patterns:

    • Define reusable abstract patterns for complex validation logic.
      xml <sch:pattern abstract="true" id="LimitNoOfWords"> <sch:rule context="$parentElement"> <sch:assert test="$words < $maxWords"> You have <sch:value-of select="$words"/> words. This should be less than <sch:value-of select="$maxWords"/>. </sch:assert> </sch:rule> </sch:pattern>

"What is the difference between conref and conkeyref in reusable content?"
"

The difference between conref and conkeyref in reusable content lies in how they reference and manage reusable elements in DITA documents:

1. Conref (Content Reference)

  • Definition: A direct reference to a specific element in another DITA file.
  • How It Works: The conref attribute points to the exact location (file path and element ID) of the reusable content.
  • Use Case: Best for scenarios where you need to reuse content from a known and fixed location.
  • Example:
    xml <p conref="topics/reusable.dita#topicid/elementid"> </p>
  • Limitation: If the file or element ID changes, the reference breaks.

2. Conkeyref (Content Key Reference)

  • Definition: An indirect reference to reusable content using a key defined in a DITA map.
  • How It Works: The conkeyref attribute uses a key defined in the DITA map to point to the reusable content. The key resolves to the actual file and element ID at runtime.
  • Use Case: Ideal for managing reusable content in large projects where content locations may change.
  • Example:
    xml <p conkeyref="keyname"> </p>
  • Advantage: If the content location changes, you only need to update the key in the DITA map, not the individual references.

Key Differences

Feature Conref Conkeyref
Reference Type Direct (file path + element ID) Indirect (key in DITA map)
Flexibility Less flexible More flexible
Maintenance Requires updating all references if content moves Update only the key in the DITA map
Use Case Small projects with fixed content locations Large projects with dynamic content locations

"How can I use key references to manage reusable content dynamically?"
"

To use key references (keyrefs) for managing reusable content dynamically in AEM Guides, follow these steps:

1. Define Keys in a DITA Map

  • Open your DITA map in the AEM Guides Editor.
  • Define keys in the <map> element to associate a key name with a specific reusable content file or element.
  • Example:
    xml <map> <topicref keys="product-description" href="topics/product-description.dita"/> </map>
    • Here, product-description is the key that points to the product-description.dita file.

2. Use Key References in Your Content

  • In your DITA topic, use the keyref attribute to reference the key defined in the DITA map.
  • Example:
    xml <p keyref="product-description"> </p>
    • This dynamically pulls the content associated with the product-description key.

3. Update Content Dynamically

  • If the location or source of the reusable content changes, update the href in the DITA map for the corresponding key.
  • Example:
    xml <map> <topicref keys="product-description" href="new-location/product-description.dita"/> </map>
    • All references to product-description will now point to the new location without needing to update individual topics.

4. Benefits of Using Key References

  • Dynamic Management: Easily update content sources by modifying the DITA map.
  • Centralized Control: Manage reusable content centrally in the DITA map.
  • Scalability: Ideal for large projects with multiple references to the same content.

"What are the limitations of reusable content in AEM Guides?"
"

Limitations of Reusable Content in AEM Guides

Reusable content in AEM Guides is a powerful feature, but it comes with certain limitations:

1. Dependency on UUIDs

  • Reusable content is linked using UUIDs. If the administrator has enabled UUID-based links, the UUID is displayed in the Link property. Renaming or moving the referenced file does not break the link, but it may confuse users expecting relative paths.

2. Embedding Breaks Dynamic Links

  • If reusable content is embedded by selecting Replace Reference with Content, the dynamic link is broken. Updates to the original content will no longer reflect in the embedded version.

3. Context Sensitivity

  • The Insert Reusable Content dialog is context-sensitive, displaying only content valid for the current location in the document. This restricts insertion in certain contexts.

4. Limited Editing Options

  • Reusable content cannot be directly edited within the current document. Authors must navigate to the original file to make changes.

5. Key Reference Dependency

  • For key-based reusable content, the DITA map containing key definitions must be properly configured. Missing or incorrect key definitions prevent insertion.

6. Search and Filtering Challenges

  • Searching for reusable content relies on UUIDs or file paths. If these identifiers are unclear, locating the desired content can be difficult.

7. Insertion Location Restrictions

  • Adding content before or after the referred content requires specific keyboard shortcuts (Alt + Left Arrow or Alt + Right Arrow), which may be unintuitive for new users.

8. Administrative Configuration

  • The ability to insert reusable content and display UUIDs or relative paths depends on administrative settings. Improper configurations can limit functionality.

9. Impact on Output

  • Reusable content may affect the final output format, especially if the referenced content is incompatible with the output type (e.g., PDF, HTML).
